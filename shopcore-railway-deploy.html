01 Environment-Driven Configuration Abstraction Both Platforms
ℹ
Industry standard: use ASPNETCORE_ENVIRONMENT + appsettings.{env}.json + environment variable overrides. Never hardcode platform-specific services. Inject via interfaces.
Create platform profile config files
appsettings.Railway.json + appsettings.Azure.json alongside base appsettings.json
// appsettings.Railway.json { "Platform": "Railway", "Storage": { "Provider": "Local" }, "Database": { "Provider": "Postgres" }, "Telemetry": { "Provider": "OpenTelemetry" } }
ConfigCritical
Define IStorageProvider interface & register by config
LocalStorageProvider and AzureBlobStorageProvider both implement IStorageProvider. DI selects at startup.
Architecture
Define ISecretProvider interface (KeyVault vs Env vars)
EnvironmentSecretProvider reads from env vars. AzureKeyVaultProvider uses Azure SDK. Swap via DI on startup based on Platform setting.
Security
Define ITelemetryProvider (AppInsights vs OpenTelemetry)
Register ApplicationInsightsTelemetryWorkerService on Azure, OpenTelemetry + OTLP exporter on Railway.
Observability
Set ASPNETCORE_ENVIRONMENT=Railway in Railway service variables
This tells the host builder which appsettings file to layer on top of the base config automatically.
Railway
02 Database — SQL Server → PostgreSQL Railway Only
⚠
Railway has no managed SQL Server. Add a PostgreSQL plugin and migrate EF Core to use Npgsql. Keep SQL Server support on Azure via provider switching.
Add PostgreSQL service in Railway dashboard
Railway → New → Database → PostgreSQL. Railway auto-injects DATABASE_URL env var.
Railway
Add Npgsql.EntityFrameworkCore.PostgreSQL NuGet package
Keep Microsoft.EntityFrameworkCore.SqlServer for Azure builds. Both can coexist.
NuGet
Register DbContext conditionally based on Database:Provider config
// Program.cs var dbProvider = builder.Configuration["Database:Provider"]; if (dbProvider == "Postgres") services.AddDbContext<AppDbContext>(o => o.UseNpgsql(config["DATABASE_URL"])); else services.AddDbContext<AppDbContext>(o => o.UseSqlServer(config["ConnectionStrings:Default"]));
Code
Run new EF Core migration for Postgres provider
dotnet ef migrations add InitPostgres --context AppDbContext. Validate no SQL Server-specific functions are used.
Migration
Apply migrations on startup (not in CI/CD)
Call dbContext.Database.MigrateAsync() in startup for Railway. Consider a separate migration job for production Azure.
Safety
03 Storage — Azure Blob → Local / S3-compatible Railway Only
✗
Railway volumes are ephemeral between redeploys. For persistent file storage on Railway, use an S3-compatible service (Cloudflare R2, Backblaze B2, MinIO). Local disk is only safe for temp files.
Implement IStorageProvider with Local + Azure + S3 variants
LocalStorageProvider (dev/test), AzureBlobStorageProvider, S3StorageProvider (Railway prod). All same interface.
Architecture
Add AWSSDK.S3 or Minio NuGet package for S3-compatible storage
Cloudflare R2 is S3-compatible and has no egress fees — good Railway pairing. Alternatively Backblaze B2.
NuGet
Set Storage:Provider in Railway env vars + add S3 credentials
STORAGE_PROVIDER=S3, S3_ENDPOINT, S3_ACCESS_KEY, S3_SECRET_KEY, S3_BUCKET — all as Railway service variables.
RailwaySecrets
Confirm no direct Azure.Storage.Blobs calls outside the provider abstraction
Search codebase for BlobServiceClient used outside IStorageProvider implementation. Move any stragglers.
Audit
04 Secrets — Azure Key Vault → Railway Variables Railway Only
List all secrets currently in Azure Key Vault
Export: connection strings, API keys, signing keys, SMTP credentials, payment gateway keys, JWT secret.
Security Audit
Add all secrets as Railway service variables (not shared variables)
Railway → Service → Variables. Use the same key names your app reads from IConfiguration so no code changes needed.
Railway
Wrap Key Vault access behind ISecretProvider so it's skipped on Railway
// Program.cs if (platform == "Azure") builder.Configuration.AddAzureKeyVault(vaultUri, credential); // On Railway: env vars are already in IConfiguration — no extra step
Code
Rotate all secrets before go-live (treat Railway as a new environment)
Never copy prod Azure secrets as-is. Generate new credentials for the Railway deployment.

05 Telemetry — App Insights → OpenTelemetry Railway Only
Add OpenTelemetry NuGet packages
OpenTelemetry, OpenTelemetry.Exporter.OpenTelemetryProtocol, OpenTelemetry.Extensions.Hosting, OpenTelemetry.Instrumentation.AspNetCore
NuGet
Choose an OTLP-compatible backend (Grafana Cloud free tier, Honeycomb, Seq)
Grafana Cloud has a generous free tier. Honeycomb is excellent for traces. Both accept OTLP. Set OTEL_EXPORTER_OTLP_ENDPOINT + OTEL_EXPORTER_OTLP_HEADERS as Railway vars.
Infra Decision
Register telemetry conditionally in Program.cs
if (platform == "Azure") services.AddApplicationInsightsTelemetry(); else services.AddOpenTelemetry() .WithTracing(b => b.AddAspNetCoreInstrumentation() .AddOtlpExporter()) .WithMetrics(b => b.AddAspNetCoreInstrumentation() .AddOtlpExporter());
Code
Verify ILogger<T> still works — Railway streams logs to dashboard natively
Railway captures stdout/stderr. Structured JSON logs (Serilog or built-in) are searchable in the Railway log viewer.

06 Dockerfile & Railway Service Setup Railway Only
Create multi-stage Dockerfile (build + runtime)
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build WORKDIR /src COPY . . RUN dotnet publish -c Release -o /app FROM mcr.microsoft.com/dotnet/aspnet:8.0 WORKDIR /app COPY --from=build /app . ENTRYPOINT ["dotnet", "Shopcore.Api.dll"]
Docker
Set PORT env var binding — Railway assigns dynamic port via $PORT
In Program.cs: builder.WebHost.UseUrls($"http://0.0.0.0:{Environment.GetEnvironmentVariable("PORT") ?? "8080"}"); or use ASPNETCORE_URLS Railway variable.
Critical
Add railway.json or use Dockerfile detection
Railway auto-detects Dockerfile. Optionally add railway.json to pin build settings, healthcheck path, restart policy.
Railway
Add a /health or /healthz endpoint for Railway healthcheck
app.MapHealthChecks("/health"). Railway uses this to determine when the container is ready and to restart on failure.
Reliability

Railway provides a *.railway.app subdomain by default. Add custom domain in Settings → Networking. TLS is automatic.
